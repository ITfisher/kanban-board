# 表设计重审 - 结构诊断与建议

## Summary

- 目标：基于当前代码与表结构，对项目数据库设计进行全面体检，输出一份偏重重构方向与建模取舍的建议稿，而非直接进入实现。
- 约束：不使用数据库外键；数据库偏存储层；关联、一致性、级联删除、同仓库校验等规则由应用代码保证。
- 当前已确认的偏好：
  - 任务负责人以用户关系表作为唯一真相来源。
  - `service_branch_stage_snapshots` 倾向尽量去掉，不再保留重型物化快照。
  - 若 `events`、`merge_operations`、`sync_runs` 没有实际业务读取价值，则倾向直接删除。
  - 优先新结构，不以平滑兼容旧 SQLite 数据为第一目标。

## Current State Analysis

### 代码中实际存在的建模特征

- 表定义分散在 `lib/db.ts` 和 `lib/schema.ts`：
  - `lib/db.ts` 负责启动建表、补列、重建旧表、兼容历史数据。
  - `lib/schema.ts` 负责 Drizzle schema 类型映射。
- 当前数据库明确关闭了 `foreign_keys`，并通过应用代码手动维护引用完整性。
- 大量聚合读取并未走数据库 JOIN，而是在 `lib/domain-data.ts` 中一次性读出多张表后在内存里拼装。
- 服务主视图严重依赖 `service_branch_stage_snapshots`，其刷新逻辑集中在 `lib/service-pipeline.ts`。
- 审计类表 `events`、`merge_operations`、`sync_runs` 当前主要由 `lib/service-pipeline.ts` 写入，没有发现页面或业务读路径消费它们。

### 已识别的结构问题

1. 单一事实来源不清晰

- `tasks` 同时保存 `owner_user_id`、`assignee_name`、`assignee_avatar`。
- `task_assignments` 也保存任务与用户关系，并由 `syncTaskOwnerAssignment()` 再做同步。
- 这导致“负责人”至少有两套来源，当前代码里已经出现双写和兜底读取。

2. 读模型与事实模型耦合过深

- `service_branch_stage_snapshots` 保存了阶段状态，也冗余了任务标题、任务状态、分支名、开发者 ID/姓名、PR 展示字段。
- 这张表承担了页面查询优化、阶段状态缓存、展示字段冗余三种职责，刷新逻辑复杂。
- 当业务规则调整时，快照表往往比事实表更难维护。

3. 审计层疑似过度设计

- `events`、`merge_operations`、`sync_runs` 都在写，但没有发现业务主链路读取。
- 三者之间存在语义重叠：
  - `events` 是广义事件流。
  - `merge_operations` 是窄化后的操作日志。
  - `sync_runs` 是同步任务运行记录。
- 如果没有 UI 或运维分析场景，当前更像写入成本和维护成本，而不是核心价值。

4. 关联表约束依赖应用层，但规则没有被系统化

- 这与“不用外键”的约束并不冲突，但当前问题在于规则散落在 API、写模型函数和同步逻辑中。
- 例如：
  - 删除任务分支时要手动删服务关联、开发者、PR、快照、审计记录。
  - 需求分支与服务必须同仓库，这一规则只在写路径里做校验。
- 现状是“应用层约束”成立，但还没有形成稳定的聚合边界与统一写入口。

5. 存储结构和查询策略存在错位

- 项目并没有真正利用数据库去做复杂查询，而是偏向“全量读表 -> 代码组装”。
- 在这种模式下，某些为 SQL 查询优化准备的冗余结构价值会下降，反而会放大同步复杂度。

## Proposed Changes

### 一、明确核心业务事实表

建议保留为核心事实模型的表：

- `repositories`
- `users`
- `tasks`
- `task_comments`
- `task_assignments`
- `task_branches`
- `task_branch_developers`
- `services`
- `service_stages`
- `task_branch_services`
- `pull_requests`
- `scm_connections`
- `repository_connections`

建议方向：

- `tasks` 只保留任务自身信息，不再持有负责人展示冗余。
- `task_assignments` 收敛为“任务与用户”的唯一关系来源。
- 若当前阶段只支持单负责人，则由应用层保证每个任务最多一条 `role = owner` 的关系。

### 二、任务负责人模型收敛

建议：

- 保留 `task_assignments`，作为任务人员关系唯一来源。
- 删除或废弃 `tasks.owner_user_id`、`tasks.assignee_name`、`tasks.assignee_avatar`。
- 任务列表/详情需要展示负责人时，统一从关系表 + 用户表映射。

原因：

- 与你的偏好一致，能消除双写与兜底逻辑。
- 即使当前只需要单负责人，也比继续把“人员关系”和“展示快照”混在 `tasks` 里更清晰。

### 三、弱化或移除 `service_branch_stage_snapshots`

建议优先方案：

- 删除 `service_branch_stage_snapshots` 作为持久化表。
- 服务主视图改为基于以下事实表实时组装：
  - `task_branch_services`
  - `task_branches`
  - `tasks`
  - `task_branch_developers`
  - `service_stages`
  - `pull_requests`

实现思路：

- 阶段状态不再落库为快照，而是通过一套纯函数按事实数据实时计算。
- 现有 `lib/service-pipeline.ts` 中的 `computeStageGateState()` 可以继续保留，但只做派生计算，不负责持久化快照。

适用前提：

- 当前数据规模不大，且页面查询大多已由代码内存拼装完成，这使“去快照”在这个项目里是现实可行的。

### 四、删除无实际消费价值的审计表

初步建议：

- 删除 `events`
- 删除 `merge_operations`
- 删除 `sync_runs`

依据：

- 当前只看到写路径，没有发现页面展示、业务决策或其他内部读取依赖。
- `sync_runs` 自身会更新自身状态，但这仍属于“写入闭环”，不是业务消费。
- 如果后续确实需要排障，再考虑以更轻量的日志方案补回，而不是先保留三张常驻业务表。

### 五、保留无外键策略，但把约束上收为“统一写模型边界”

建议：

- 不引入数据库外键。
- 不再把约束散在多个 API 路由中。
- 将以下规则集中到领域写入层：
  - 任务负责人唯一性
  - 任务分支与服务必须同仓库
  - 删除任务/分支/服务时的手动级联清理
  - PR 创建的业务前置条件

建议承载位置：

- 延续 `lib/task-branch-write.ts`、`lib/task-assignment-write.ts` 这类写模型模块的方向。
- 后续可继续抽出服务、PR、仓库删除等统一写入入口，减少 API 层直接拼表操作。

## Assumptions & Decisions

### 已确认决策

- 不使用外键。
- 数据一致性由代码保证。
- 任务负责人采用关系表作为唯一真相来源。
- 当前任务人员关系只要求“单负责人”。
- 快照表倾向删除，而不是继续加强。
- 审计表若无实际读取价值，则直接删除。

### 当前关键判断

- 现有项目更像“前端应用 + 本地 SQLite 存储”，而不是依赖数据库关系能力的重后端系统。
- 因此表结构应优先围绕“事实最少、职责清晰、应用层好维护”来设计，而不是围绕 SQL 层复杂优化来设计。

## Verification Steps

后续继续讨论时，重点按下面顺序验证：

1. 逐张确认哪些表是“事实表”，哪些是“派生表”。
2. 明确每个聚合的唯一写入口，避免双写。
3. 针对计划删除的表，逐一确认：
   - 有没有页面读取
   - 有没有 API 返回依赖
   - 有没有内部逻辑把它们当作判断依据
4. 为“去快照”后的服务主视图列出实时组装所需字段与计算过程。
5. 为“单负责人关系表”定义最小可行的数据约束与应用层守卫规则。
