# 新项目重构方案：研发效能管理系统

## Summary

目标：以 `design.txt` 为目标蓝图，将当前项目从“任务下挂服务分支状态”的实现，重构为“任务 -> 需求分支 -> 服务阶段流水线 -> PR/同步事件”的新项目架构。

本次方案按新项目视角设计：

* 不兼容历史数据与旧接口。

* 允许直接重建数据库 schema、API 资源边界、页面数据流。

* 保留当前技术栈思路：Next.js App Router + Route Handler + SQLite + Drizzle。

已确认的关键决策：

* 阶段模型：每个服务自定义阶段流水线。

* 分支归属：需求分支归仓库，再挂多个服务。

* 用户模型：先建设 `users`，暂不建设 `teams`，也先不做复杂 RBAC。

* 历史留存：保留完整历史，包括每次合并、撤销合并、轮询/同步结果、状态变更。

* 协作模型：任务有负责人，分支可挂多个开发用户，两者分开统计。

## Current State Analysis

基于仓库现状梳理：

### 已有能力

* 当前项目已有服务端持久化边界：`app/api/*`。

* 当前项目已有 SQLite + Drizzle 基础设施：`lib/db.ts`、`lib/schema.ts`。

* 当前项目已有 GitHub 代理与状态查询：`app/api/github/*`。

* 当前项目已有任务、服务、分支、设置等页面骨架：`app/tasks/*`、`app/services/*`、`app/branches/page.tsx`、`app/settings/page.tsx`。

### 当前模型的主要问题

现有实现主要围绕以下表：

* `tasks`

* `services`

* `service_branches`

* `github_configs`

* `settings`

核心问题：

* `service_branches` 实际是“任务下某个服务的一条分支记录”，不是 `design.txt` 中的“需求分支主实体”。

* 当前模型是“一个服务一条分支状态”，无法表达“一个需求分支挂多个同仓库服务”。

* 当前阶段逻辑固化为 `test/master` 两段，见 `service_branches` 中的 `testPullRequestUrl`、`masterPullRequestUrl`、`mergedToTest`、`mergedToMaster`。

* 当前没有显式的 `repository` 实体，只在 `services.repository` 里保存字符串。

* 当前没有 `users`、`task assignees`、`branch developers` 等正式组织模型。

* 当前没有可审计的“PR 历史 / 合并历史 / 撤销合并历史 / 同步运行历史 / 状态事件历史”。

* 当前 GitHub 集成默认按 `serviceName -> repo slug` 推导仓库，边界不稳定，不适合 Monorepo/多服务精确建模。

### 结论

如果按 `design.txt` 真正落地，新项目不应继续在 `service_branches` 上扩展，而应重建领域模型与表结构。

## Proposed Changes

## 一、目标领域模型

新项目的核心对象建议调整为：

1. `Repository`

* Git 仓库根实体。

* 需求分支归属于仓库。

* 服务归属于仓库。

1. `Service`

* 发布流转单位。

* 同一仓库下可有多个服务。

* 每个服务独立维护阶段流水线。

1. `Task`

* 看板任务/需求条目。

* 用于管理、排期、统计负责人维度。

1. `TaskBranch`

* 需求分支主实体。

* 属于某个 `Task`，也属于某个 `Repository`。

* 可挂多个 `Service`。

* 可挂多个开发用户。

1. `ServiceStage`

* 服务的阶段定义，如 `QA` / `Pre` / `Prod`。

* 支持启停、排序、目标分支、允许的合并来源。

1. `PullRequest`

* 某个 `TaskBranch + Service + Stage` 的一次实际 PR 单据。

* 允许一个阶段重提多次 PR。

1. `Event / MergeOperation / SyncRun`

* 审计、轮询、合并与撤销合并的历史记录。

## 二、建议数据库表结构

以下为建议的新表结构。

### 1. 人员

#### `users`

* `id` TEXT PK

* `name` TEXT NOT NULL

* `email` TEXT NULL UNIQUE

* `avatar_url` TEXT NULL

* `is_active` INTEGER NOT NULL DEFAULT 1

* `created_at` TEXT NOT NULL

* `updated_at` TEXT NOT NULL

用途：

* 任务负责人

* 分支开发用户

* 操作人/审计人

### 2. 仓库与服务

#### `repositories`

* `id` TEXT PK

* `domain` TEXT NOT NULL

* `owner` TEXT NOT NULL

* `name` TEXT NOT NULL

* `clone_url` TEXT NULL

* `default_branch` TEXT NOT NULL DEFAULT 'main'

* `created_at` TEXT NOT NULL

* `updated_at` TEXT NOT NULL

约束：

* UNIQUE(`domain`, `owner`, `name`)

说明：

* 这是新模型的根实体。

* 替代当前 `services.repository` 的字符串弱引用方式。

#### `services`

* `id` TEXT PK

* `repository_id` TEXT NOT NULL

* `name` TEXT NOT NULL

* `description` TEXT NOT NULL DEFAULT ''

* `root_path` TEXT NOT NULL DEFAULT ''

* `is_active` INTEGER NOT NULL DEFAULT 1

* `created_at` TEXT NOT NULL

* `updated_at` TEXT NOT NULL

约束：

* UNIQUE(`repository_id`, `name`)

说明：

* `root_path` 对应 `design.txt` 中的 Monorepo 路径。

#### `service_dependencies`

* `service_id` TEXT NOT NULL

* `depends_on_service_id` TEXT NOT NULL

* `created_at` TEXT NOT NULL

约束：

* UNIQUE(`service_id`, `depends_on_service_id`)

用途：

* 服务依赖关系与发布门禁提示。

### 3. 服务阶段流水线

#### `service_stages`

* `id` TEXT PK

* `service_id` TEXT NOT NULL

* `key` TEXT NOT NULL

* `name` TEXT NOT NULL

* `position` INTEGER NOT NULL

* `enabled` INTEGER NOT NULL DEFAULT 1

* `target_branch` TEXT NOT NULL

* `allow_merge_from` TEXT NOT NULL DEFAULT 'feature\_or\_previous'

* `created_at` TEXT NOT NULL

* `updated_at` TEXT NOT NULL

约束：

* UNIQUE(`service_id`, `key`)

* UNIQUE(`service_id`, `position`)

字段语义：

* `key`: 稳定标识，如 `qa/pre/prod`

* `name`: 展示名，如 `QA/预发/正式`

* `position`: 顺序

* `target_branch`: 该阶段实际合并到的目标分支

* `allow_merge_from`: `feature_only | previous_only | feature_or_previous`

说明：

* 这是 `design.txt` 中 `Pipeline_Config` 的正式表结构落地。

* 如果某阶段禁用，则下游阶段自动以前一个启用阶段作为“前置阶段”。

### 4. 任务与任务负责人

#### `tasks`

* `id` TEXT PK

* `title` TEXT NOT NULL

* `description` TEXT NOT NULL DEFAULT ''

* `status` TEXT NOT NULL

* `priority` TEXT NOT NULL DEFAULT 'medium'

* `created_at` TEXT NOT NULL

* `updated_at` TEXT NOT NULL

* `completed_at` TEXT NULL

说明：

* 保留任务作为看板主实体，但不再直接内嵌分支状态。

#### `task_assignees`

* `task_id` TEXT NOT NULL

* `user_id` TEXT NOT NULL

* `role` TEXT NOT NULL DEFAULT 'owner'

* `created_at` TEXT NOT NULL

约束：

* UNIQUE(`task_id`, `user_id`, `role`)

说明：

* 用于“负责人复制任务”的统计口径。

* 允许一个任务多个负责人/参与人。

### 5. 需求分支主模型

#### `task_branches`

* `id` TEXT PK

* `task_id` TEXT NOT NULL

* `repository_id` TEXT NOT NULL

* `branch_name` TEXT NOT NULL

* `status` TEXT NOT NULL DEFAULT 'active'

* `created_at` TEXT NOT NULL

* `updated_at` TEXT NOT NULL

约束：

* UNIQUE(`repository_id`, `branch_name`)

说明：

* 这是重构后的核心主实体。

* 一个任务可有多个需求分支。

* 一个需求分支只属于一个仓库。

#### `task_branch_services`

* `task_branch_id` TEXT NOT NULL

* `service_id` TEXT NOT NULL

* `created_at` TEXT NOT NULL

约束：

* UNIQUE(`task_branch_id`, `service_id`)

规则：

* `service.repository_id` 必须等于 `task_branches.repository_id`。

* 同一需求分支可以关联该仓库下多个服务。

#### `task_branch_developers`

* `task_branch_id` TEXT NOT NULL

* `user_id` TEXT NOT NULL

* `role` TEXT NOT NULL DEFAULT 'developer'

* `created_at` TEXT NOT NULL

约束：

* UNIQUE(`task_branch_id`, `user_id`, `role`)

说明：

* 满足“具体分支有对应的开发用户”。

* 统计维度将明显优于当前仅有 assignee 字符串的方式。

### 6. PR 流转与当前状态

#### `pull_requests`

* `id` TEXT PK

* `task_branch_id` TEXT NOT NULL

* `service_id` TEXT NOT NULL

* `stage_id` TEXT NOT NULL

* `provider` TEXT NOT NULL DEFAULT 'github'

* `repo_domain` TEXT NOT NULL

* `repo_owner` TEXT NOT NULL

* `repo_name` TEXT NOT NULL

* `pr_number` INTEGER NOT NULL

* `pr_url` TEXT NOT NULL

* `head_ref` TEXT NOT NULL

* `base_ref` TEXT NOT NULL

* `title` TEXT NOT NULL

* `body` TEXT NULL

* `state` TEXT NOT NULL

* `merged` INTEGER NOT NULL DEFAULT 0

* `mergeable` INTEGER NULL

* `mergeable_state` TEXT NULL

* `opened_at` TEXT NOT NULL

* `merged_at` TEXT NULL

* `closed_at` TEXT NULL

* `created_at` TEXT NOT NULL

* `updated_at` TEXT NOT NULL

约束：

* UNIQUE(`repo_domain`, `repo_owner`, `repo_name`, `pr_number`)

说明：

* 一个阶段可以多次发起 PR，因此不建议只保留“最新 PR URL”在主表字段里。

#### `pull_request_checks`

* `pull_request_id` TEXT NOT NULL

* `head_sha` TEXT NOT NULL

* `state` TEXT NOT NULL

* `total_count` INTEGER NOT NULL DEFAULT 0

* `completed_count` INTEGER NOT NULL DEFAULT 0

* `failed_count` INTEGER NOT NULL DEFAULT 0

* `updated_at` TEXT NOT NULL

约束：

* UNIQUE(`pull_request_id`)

说明：

* 保存最新检查状态用于页面快速展示。

* 检查历史进入事件表。

#### `branch_diffs`

* `id` TEXT PK

* `task_branch_id` TEXT NOT NULL

* `service_id` TEXT NOT NULL

* `stage_id` TEXT NOT NULL

* `ahead_by` INTEGER NOT NULL DEFAULT 0

* `behind_by` INTEGER NOT NULL DEFAULT 0

* `total_commits` INTEGER NOT NULL DEFAULT 0

* `status` TEXT NOT NULL DEFAULT ''

* `updated_at` TEXT NOT NULL

约束：

* UNIQUE(`task_branch_id`, `service_id`, `stage_id`)

说明：

* 替代当前 `service_branches.diffStatus` 的 JSON blob。

#### `service_branch_stage_snapshots`

* `id` TEXT PK

* `service_id` TEXT NOT NULL

* `task_branch_id` TEXT NOT NULL

* `stage_id` TEXT NOT NULL

* `latest_pull_request_id` TEXT NULL

* `stage_status` TEXT NOT NULL

* `gate_status` TEXT NOT NULL

* `is_actionable` INTEGER NOT NULL DEFAULT 0

* `action_type` TEXT NULL

* `display_order` INTEGER NOT NULL DEFAULT 0

* `last_synced_at` TEXT NULL

* `updated_at` TEXT NOT NULL

约束：

* UNIQUE(`service_id`, `task_branch_id`, `stage_id`)

字段语义：

* `stage_status`: 当前阶段状态，建议枚举 `not_created | open | merged | closed | conflict | blocked`

* `gate_status`: 门禁状态，建议枚举 `waiting_previous_stage | ready | blocked_by_conflict | blocked_by_dependency`

* `is_actionable`: 当前是否可在服务主视角直接操作

* `action_type`: 建议枚举 `create_pr | merge_check | retry_sync | none`

说明：

* 这是服务主视角的读模型/快照表。

* 它不替代 `pull_requests` 历史表，而是为“当前这个服务下，哪些需求分支已经可以合并、现在卡在哪一步”提供高效查询。

* 该表由同步引擎或 PR 变更逻辑增量更新。

### 7. 历史、审计与状态感知

#### `events`

* `id` TEXT PK

* `type` TEXT NOT NULL

* `scope` TEXT NOT NULL

* `scope_id` TEXT NOT NULL

* `actor_user_id` TEXT NULL

* `payload_json` TEXT NOT NULL DEFAULT '{}'

* `created_at` TEXT NOT NULL

建议事件类型：

* `pr.created`

* `pr.updated`

* `pr.merged`

* `pr.closed`

* `pr.reverted`

* `sync.started`

* `sync.finished`

* `sync.failed`

* `stage.unlocked`

* `gate.blocked`

* `branch.linked_service`

* `branch.unlinked_service`

说明：

* 这是完整历史的统一沉淀层。

#### `merge_operations`

* `id` TEXT PK

* `task_branch_id` TEXT NOT NULL

* `service_id` TEXT NOT NULL

* `stage_id` TEXT NOT NULL

* `pull_request_id` TEXT NULL

* `operation` TEXT NOT NULL

* `reason` TEXT NULL

* `actor_user_id` TEXT NULL

* `created_at` TEXT NOT NULL

说明：

* 单独保留“合并 / 撤销合并”语义，方便报表与回溯。

#### `sync_runs`

* `id` TEXT PK

* `repository_id` TEXT NOT NULL

* `trigger` TEXT NOT NULL

* `status` TEXT NOT NULL

* `started_at` TEXT NOT NULL

* `finished_at` TEXT NULL

* `error_message` TEXT NULL

* `stats_json` TEXT NOT NULL DEFAULT '{}'

说明：

* 用于无 Webhook 场景下的主动轮询记录。

* 后续可以做 repo 级请求去重、限流、错误诊断。

### 8. GitHub / GHE 配置

#### `scm_connections`

* `id` TEXT PK

* `name` TEXT NOT NULL

* `provider` TEXT NOT NULL DEFAULT 'github'

* `domain` TEXT NOT NULL

* `owner` TEXT NOT NULL

* `token` TEXT NOT NULL

* `is_default` INTEGER NOT NULL DEFAULT 0

* `created_at` TEXT NOT NULL

* `updated_at` TEXT NOT NULL

说明：

* 替代当前 `github_configs`。

* token 继续仅服务端可见。

#### `repository_connections`

* `repository_id` TEXT NOT NULL

* `scm_connection_id` TEXT NOT NULL

* `created_at` TEXT NOT NULL

约束：

* UNIQUE(`repository_id`, `scm_connection_id`)

说明：

* 让“仓库”和“连接配置”显式绑定，避免当前按服务名/域名猜测仓库。

## 三、推荐页面与 API 重构方式

### 页面重构建议

#### 1. 任务页 `/tasks`

* 只负责任务管理、看板流转、负责人统计入口。

* 不直接承载服务分支的低层细节。

#### 2. 任务详情页 `/tasks/[id]`

* 展示任务信息。

* 展示该任务下的多个 `task_branches`。

* 每个需求分支下展示：

  * 所属仓库

  * 关联服务

  * 开发用户

  * 各服务各阶段的 PR 流转状态

#### 3. 服务页 `/services`

* 维护服务基本信息。

* 配置 `root_path`

* 配置 `service_stages`

* 配置服务依赖

#### 4. 服务主视角页 `/services/[id]`

* 这是你提到的“服务维度主视角”页面。

* 推荐支持两种展示模式：

  * 阶段横向展开矩阵

  * 按阶段 `Tab` 切换

* 默认推荐桌面端使用“阶段横向展开矩阵”，窄屏或移动端使用“按阶段 Tab 切换”。

* 无论哪种模式，数据来源都一致：只展示当前服务相关的需求分支。

* 每一行需要展示：

  * 任务标题

  * 需求分支名

  * 开发用户

  * 当前所处阶段

  * 当前阶段 PR 状态

  * 是否已具备“可合并 / 可提测 / 可发布”条件

  * 最近一次同步时间

* 页面需要支持的筛选：

  * `可合并`

  * `阻塞中`

  * `已合并`

  * `有冲突`

  * `按阶段`

* 如果采用阶段 `Tab`：

  * 每个 `Tab` 对应一个 `service_stage`

  * 打开某个阶段后，页面拆成两个分组：

    * `未合并需求分支`

    * `已合并需求分支`

  * `未合并需求分支` 按优先级排序建议为：

    * `可操作`

    * `冲突/阻塞`

    * `仅已创建 PR`

    * `尚未发起`

* 如果采用阶段横向展开：

  * 每一行一个 `task_branch`

  * 每一列一个阶段

  * 单元格展示该分支在该阶段的状态徽标、PR 入口、可操作按钮

  * 当前阶段列可高亮，已完成阶段弱化显示，未解锁阶段置灰

* 页面需要支持的动作：

  * 发起当前阶段 PR

  * 刷新当前分支状态

  * 跳转到 PR

  * 跳转到任务详情

* 推荐交互：

  * 顶部先放阶段 `Tab`

  * `Tab` 内容区内再提供“矩阵视图 / 列表视图”切换

  * 这样既满足你快速看“这个阶段有哪些已合并/未合并需求分支”，也保留完整阶段轨迹查看能力

#### 5. 分支页 `/branches`

* 改成真正的“分支主视角”

* 一行一个 `task_branch`

* 横向展示其挂靠服务在各阶段的状态矩阵

#### 6. 新增仓库页 `/repositories`

* 维护仓库

* 绑定 SCM 配置

* 查看该仓库下服务与分支

#### 7. 新增用户页

* 用户管理

* 统计口径准备

### API 资源建议

建议新建或重组为以下资源：

* `/api/repositories`

* `/api/services`

* `/api/services/[id]/stages`

* `/api/services/[id]/branch-pipeline`

* `/api/services/[id]/stage-board`

* `/api/users`

* `/api/tasks`

* `/api/tasks/[id]/assignees`

* `/api/task-branches`

* `/api/task-branches/[id]/services`

* `/api/task-branches/[id]/developers`

* `/api/task-branches/[id]/pull-requests`

* `/api/pull-requests/[id]/sync`

* `/api/sync/repositories/[id]`

* `/api/scm-connections`

GitHub 代理层建议保留，但入参改为显式仓库信息，而不是当前偏依赖 `serviceName`：

* 传 `repository_id` 或 `repo_domain/owner/name`

* 不再用 `toRepoSlug(serviceName)` 这种推导方式作为主路径

工作内容：

* 统一封装 repo 级去重轮询。

* 进入任务详情页、服务页、分支页时触发按仓库同步。

* 每次同步写入 `sync_runs` 与 `events`。

* 当阶段合并完成时：

  * 更新 PR 状态

  * 写入 `merge_operations`

  * 写入 `events`

  * 自动判断并解锁下一个启用阶段

* 同步完成后，刷新 `service_branch_stage_snapshots`，用于服务主视角页面快速查询

## 五、关键规则设计

### 阶段门禁

建议规则：

* Stage N 发起前，Stage N-1 的所有关联服务 PR 必须已 `MERGED`。

* 如果某服务的中间阶段被禁用，则自动向前寻找最近一个启用阶段作为前置阶段。

* `allow_merge_from` 决定创建 PR 时的 source branch：

  * `feature_only`：只能从需求分支发起

  * `previous_only`：只能从上一个环境分支发起

  * `feature_or_previous`：由用户选择

### 统计口径

建议拆成三类：

* 任务负责人统计：来自 `task_assignees`

* 分支开发统计：来自 `task_branch_developers`

* 服务/阶段交付统计：来自 `pull_requests + merge_operations + events`

* 服务主视角当前状态：优先来自 `service_branch_stage_snapshots`

### 审计口径

必须记录：

* 谁创建了 PR

* 谁执行了合并/撤销合并

* 合并前后状态

* 轮询结果与错误

* 阶段解锁/阻塞原因

## 六、与当前架构的对应关系

按仓库现状，新旧关系建议如下：

* 旧 `services.repository`
  -> 新 `repositories` + `services.repository_id`

* 旧 `service_branches`
  -> 新 `task_branches`
  -> 新 `task_branch_services`
  -> 新 `pull_requests`
  -> 新 `branch_diffs`

* 旧 `github_configs`
  -> 新 `scm_connections` + `repository_connections`

* 旧任务里的 `assignee`
  -> 新 `task_assignees`

* 当前页面里的“测试/线上 PR”
  -> 新的按 `service_stages.position` 驱动的阶段流水线

* 当前服务页只做配置管理
  -> 新增 `services/[id]` 作为服务维度主视角

## 七、Assumptions & Decisions

已锁定的假设：

* 一个任务可以拥有多个需求分支。

* 一个需求分支只属于一个仓库。

* 一个需求分支可以挂多个同仓库服务。

* 一个服务可以配置任意数量的阶段。

* 一个阶段可以多次创建 PR，因此 PR 必须单独建表。

* 新项目允许直接舍弃旧 schema 与旧接口兼容。

当前仍建议继续讨论但不阻塞建模的点：

* `task_assignees.role` 是否要严格区分 `owner` / `pm` / `qa` / `participant`

* 是否需要 `milestones/sprints` 维度做进一步统计

* 是否需要在 `repositories` 层支持默认阶段模板，再下发到服务

## Verification

方案确认后，执行阶段建议按以下方式验证：

1. Schema 验证

* 能创建仓库、服务、阶段、任务、需求分支、PR、事件等完整链路

* 能约束“分支只能关联同仓库服务”

1. 业务流验证

* 创建一个任务

* 在同一仓库下创建一个需求分支

* 该分支挂多个服务

* 各服务按自定义阶段发起 PR

* 合并后自动解锁下一阶段

* 撤销合并后产生记录

1. 状态同步验证

* 进入详情页触发 repo 级同步

* 多个同仓库服务请求可以去重

* 同步结果正确写入 `sync_runs` 和 `events`

1. 前端验证

* 任务详情页能正确展示“任务 -> 分支 -> 服务 -> 阶段”

* 分支页能以矩阵方式展示多服务阶段进度

* 用户维度统计可区分“任务负责人”和“分支开发者”

1. 工程验证

* `pnpm lint`

* `pnpm typecheck`

* 若引入新测试，则至少覆盖 schema mapper 与阶段门禁逻辑
